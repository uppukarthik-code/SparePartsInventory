## 2. Pre-flight findings (blockers discovered before implementation)

Two issues had to be resolved before activation could produce real division outputs. **Neither involved analytical code.**

### 2.1 Missing operational input — pipeline was broken
The entire operational pipeline reads a single workbook: `cfg.OPERATIONAL_WORKBOOK = raw_data/railway_stock_summary.xlsx`. That file **was absent from disk** at implementation time (it existed earlier in the session and had since been removed; it is untracked in git and generated by no code). The runner crashed for **every** division with:
```
load_operational_stock() -> _xlsx_rows_via_xml(cfg.OPERATIONAL_WORKBOOK)
FileNotFoundError: ...raw_data\railway_stock_summary.xlsx
```

### 2.2 Per-division data not wired into the operational source
The runner partitions **one** consolidated operational frame by depot→BU. The five divisions' data lives in **separate** files the runner never reads: `raw_data/Railway_Operations/<BU>/stock_history.xlsx`. Without restoring the consolidated workbook, split-mode would have yielded 0 rows → skeleton markers, not real analytics.

**Decision (user-approved):** *Rebuild the operational workbook from the six per-division files, then activate.* This is a data-restore/ingestion step, not an architecture change.

## 5. Important data-model note (honest characterisation)

The pipeline has **two universes** (per the Step-18 assessment), and activation behaves differently in each:

- **Operational universe** (depot-granular) → **genuinely partitioned per division.** Operational inventory, aging, dead/slow stock, valuation, operational ABC, and the `op_*` Power BI pages differ per division (table 4.2).
- **Strategic universe** (`railways.xlsx`, 59 zone-level vital items) → **shared across all divisions by design.** Demand history, forecast, inventory policy, SKU master, SRRS, and procurement KPIs are driven by these 59 zone items, so they are **identical** across all six divisions. This is the documented zone-consolidated model, not a defect — division-level strategic splitting was explicitly out of scope.

**Consequence:** each division's *procurement / forecasting / policy* outputs are the shared zone analysis; each division's *operational health* outputs are its own. This is reflected faithfully in the generated files.
